# 선언형 UI (Declarative UI)

React가 **왜 존재하는가**에 대한 답. 실습: [counter-app](../../projects/counter-app)

## 코드 근거

- `Viewer.tsx`: `<p>{ count }</p>` — 화면에 DOM 조작 코드가 한 줄도 없다.
- 바닐라 JS라면 카운트를 바꿀 때: `element.textContent = count` 처럼 DOM을 직접 지시해야 한다.

## 정리

> **React의 세계관: 화면 = f(상태).** 컴포넌트는 "현재 상태를 넣으면 화면 생김새(JSX)를 뱉는 함수"다. 이 한 줄이 선언형의 뿌리다.

### 1. 명령형 vs 선언형

**명령형(imperative)** — "어떻게 바꿀지"를 단계별로 지시. 상태와 화면을 내 손으로 동기화.

```js
count += 1
document.querySelector('#count').textContent = count  // 내가 직접 DOM 갱신
```

**선언형(declarative)** — "무엇을 보여줄지"만 선언. 동기화는 React가.

```tsx
<p>{count}</p>       // "count면 이렇게 보여라" 선언
setCount(count + 1)  // 상태만 바꿈. DOM은 안 건드림
```

내 코드엔 DOM 조작이 없는데 화면이 바뀌는 이유 = 그 일을 **React가** 대신 하기 때문.

### 2. 선언형이 왜 좋은가 (핵심)

`count`를 화면 N군데(숫자, 짝/홀 라벨, 경고문, 탭 제목…)에 쓴다고 하자.

- **명령형**: 상태를 바꿀 때마다 그 N군데를 내가 수동으로 동기화. 하나라도 빠뜨리면 상태-화면 불일치(버그). N이 커질수록 실수가 폭발.
- **선언형**: 각 위치에 "count면 이렇게"만 선언 → `setCount` 한 번에 React가 N군데를 일관되게 맞춤. **불일치 버그가 원천적으로 안 생긴다.**

→ 선언형은 **동기화 책임을 나 → React로** 넘기는 것. 이게 React가 존재하는 이유.

### 3. React는 그걸 어떻게 빠르게 하나 (Virtual DOM)

리렌더는 2단계다:

1. **렌더(재실행)** — 컴포넌트 함수를 다시 실행. 결과는 실제 DOM이 아니라 "화면이 이래야 한다"는 가벼운 JS 객체 트리 = **Virtual DOM**. (만드는 비용이 쌈)
2. **커밋** — 새 VDOM을 직전 VDOM과 **비교(reconciliation)** → **달라진 부분만** 실제 DOM에 최소로 반영.

따라서:

- **전부 리렌더(함수 재실행) ≠ 전부 DOM 갱신.** 실제 DOM은 diff로 걸러진 최소한만 바뀐다.
- 핵심 아이디어: 실제 DOM 조작이 비싸니, 값싼 JS 객체(VDOM)끼리 먼저 비교해서 필요한 변경만 추린다.
- ⚠️ "VDOM = 리렌더 공짜"는 아니다. 재실행 자체도 비용이 있어 불필요한 리렌더를 줄이는 `React.memo` 최적화가 따로 있다. (→ [rendering-model](./rendering-model.md))

### 4. (적용) 선언형의 예외 — 명령형이 필요할 때

판별 기준(**화면 = f(상태)** 관점): 버튼(이벤트)이 하는 일이

- **상태(값)를 바꾸는 것** → 선언형. 화면은 `f(상태)`로 자동 반영. (예: 좋아요 → `liked=true`, count +1)
- **화면 생김새로 표현 안 되는 동작(행위)을 시키는 것** → 명령형. 브라우저/DOM에 직접 시켜야 함. (예: 스크롤)

> ⚠️ 트리거가 "클릭 한 번"이냐는 기준이 아니다(둘 다 클릭이다). 기준은 **결과가 "상태 값"이냐 "행위"냐**.
> 리트머스: "이 동작의 결과를 **화면에 그려지는 '값' 하나로 적을 수 있나?**" → 적을 수 있으면 상태(선언형) / 브라우저·요소한테 "~해"라고 시켜야 하면 동작(명령형).

명령형이 필요한 예 (모두 "값"이 아니라 "행위"):

- 검증 실패 시 에러 입력칸으로 커서 이동, 특정 위치로 스크롤
- 비디오 재생/정지, 요소 크기 측정
- 외부 라이브러리(지도·차트 등)가 자기 DOM을 직접 관리

이때의 탈출구가 **ref (`useRef`)**: `<input ref={inputRef}/>` 후 `inputRef.current.focus()`처럼 실제 DOM 요소에 직접 명령한다. (참고: 마운트 시 포커스처럼 상태로 표현 가능한 건 `autoFocus` prop 같은 선언형이 우선. 자세한 건 나중 ref 노트)

## 한 줄 요약

선언형 = "상태 → 화면 생김새"만 선언하면 동기화는 React가 담당(VDOM diff로 바뀐 부분만 실제 DOM 반영). 기본은 선언형(상태 바꾸기), 상태 값으로 표현 안 되는 동작이 필요한 예외에서만 `useRef`로 직접 DOM을 만진다.
