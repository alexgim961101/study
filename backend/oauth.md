# OAuth 2.0 (Authorization Code + PKCE)

내 비밀번호를 넘기지 않고, 제3자 앱에게 **"좁은 권한의 임시 토큰"만** 내줘서 내 데이터에 접근하게 하는 **위임 인가(delegated authorization)** 프로토콜.
관련 실습/코드: 없음 (포토프린팅 ↔ 구글 포토 시나리오로 정리)

## 예시 근거

- **포토프린팅**(제3자 앱)이 내 **구글 포토** 사진을 가져와 인화한다. 그러려면 내 구글 포토에 접근해야 한다.
- 대원칙: **포토프린팅은 내 구글 비밀번호를 끝까지 모른다.** 로그인은 오직 google.com 화면에서만 일어난다.

## 정리

### 1. 왜 필요한가 — 비밀번호를 "직접" 넘기면 생기는 3대 문제

OAuth가 없던 시절처럼 포토프린팅에 **구글 아이디·비번을 그냥 넘겨주면** 세 가지가 터진다.

| # | 문제 | 내용 |
|---|------|------|
| 1 | **범위(scope)** | 비번은 계정 전체의 열쇠. "포토 읽기"만 주고 싶어도 Gmail·드라이브까지 통째로 넘어간다 (all-or-nothing). |
| 2 | **개별 취소 불가** | 여러 서비스에 같은 비번을 넘겼다면, 하나만 끊으려 비번을 바꿔도 **전부 다 끊긴다.** |
| 3 | **자격증명 노출** | 상대가 내 진짜 비번을 저장한다. 그 회사가 털리면 **내 구글 계정도 털린다.** 2단계 인증도 무력화. |

세 문제의 **공통 원인은 하나** — "비밀번호를 직접 넘겼다"는 것.

### 2. 해법: 비밀번호 대신 토큰

공유되는 비밀번호 하나 → **앱마다 발급되는 좁은 권한의 토큰**으로 바꾸면 세 문제가 동시에 풀린다.

- **액세스 토큰(access token)**: 실제 요청 때 제시하는 임시 출입증.
  - **좁은 범위** → #1 해결 + 유출돼도 피해 반경(blast radius)이 작음(#3 완화)
  - **앱마다 별도 발급 → 개별 폐기 가능** → #2 해결
  - **짧은 수명(만료)** → 유출돼도 악용 창(window)이 짧음(#3 완화)
- **리프레시 토큰(refresh token)**: 액세스 토큰이 만료되면 **재로그인 없이** 새 액세스 토큰을 받아오는 용도.

| | 액세스 토큰 | 리프레시 토큰 |
|---|---|---|
| **제시 대상** | Resource Server (구글 포토 API) | Authorization Server(발급처)에만 |
| **용도** | 데이터 접근(사진 읽기) | 새 액세스 토큰 발급 **전용**(데이터 접근 불가) |
| **형식(흔히)** | 자체 완결형(JWT), 서명 검증 | 불투명 문자열, 발급처가 DB 조회 |
| **폐기** | 만료 전엔 어려움 → 그래서 짧게 | 개별 폐기(revoke) 가능 |
| **노출 빈도** | 자주(요청마다) | 드묾(서버에 보관, 발급처에만 감) |
| **수명** | 짧음(분~시간) | 긺(일~월) — 위 특성들의 *결과* |

핵심: **수명 차이는 원인이 아니라 결과다.** 제시 대상·용도가 다르니 → 노출·폐기 특성이 갈리고 → 그래서 액세스 토큰은 짧게, 리프레시 토큰은 길게 만든다.

> 리프레시 토큰 저장 ≠ 비밀번호 저장. 유출돼도 (1) 범위가 좁고 (2) **그 토큰 하나만 폐기**하면 되며(비번은 바꾸면 전부 끊김) (3) 그걸로 비번 변경·계정 탈취는 못 한다.

### 3. 앞문 / 뒷문과 인가 코드 — 토큰을 안전하게 전달하기

로그인 성공 후, 구글은 토큰을 포토프린팅에 어떻게 전달할까? 이 순간 둘을 잇는 통로는 **내 브라우저**뿐이다. 통로를 둘로 나눠 본다.

- **앞문 = 브라우저**: 구글 → 브라우저 → 포토프린팅 리다이렉트. 주소(URL)·방문 기록·서버 로그·리퍼러에 **흔적이 남아 위험.**
- **뒷문 = 서버-서버 직통**: 포토프린팅 서버가 구글 서버에 직접 호출. 브라우저를 안 거쳐 **흔적이 안 남음.**

그래서:
- **앞문엔 진짜 토큰을 실으면 안 된다.** 대신 **일회용 교환권 = 인가 코드(authorization code)** 만 보낸다. (짧은 수명 + 1회용)
- **진짜 토큰(액세스·리프레시)은 뒷문에서만** 주고받는다. 포토프린팅 서버가 인가 코드를 토큰으로 **교환(exchange)** 한다.
- 뒷문 교환 때 포토프린팅은 자신이 진짜임을 증명한다. 서버 앱은 **client secret**(앱의 비밀키)으로, 비밀키를 못 숨기는 앱은 **PKCE**로(→ 4).

### 4. PKCE — client secret을 못 숨기는 앱을 위한 방어

SPA(브라우저 전용)·모바일 앱은 **client secret을 안전하게 숨길 데가 없다**(JS·앱 파일을 뜯으면 노출). 그래서 미리 정해둔 고정 비밀 대신 **요청마다 즉석에서 만드는 일회용 비밀**을 쓴다.

- **열쇠(code_verifier)**: 로그인 시작 때 앱이 즉석 생성한 랜덤 비밀. **세션마다 다르고**, 앱 주머니에만 있고 네트워크로 안 나간다.
- **자물쇠(code_challenge)**: 열쇠를 해시한 값(역산 불가). authorization 요청(앞문)에 실어 구글에 **미리 맡긴다.**
- 구글은 **발급할 인가 코드에 자물쇠를 묶어** 보관한다.
- 뒷문 교환 때 앱이 **원본 열쇠**를 제시 → 구글이 해시해서 맡아둔 자물쇠와 **대조** → 일치하면 토큰 발급.

**왜 안전한가 (공격 시나리오로 검증):** 공격자가 인가 코드를 훔쳐도 못 바꾼다.
- 구글에 직접 교환 시도 → 서버 앱은 **client secret**이 없어서 실패.
- 훔친 코드를 포토프린팅에 자기 세션으로 들이밀어도 → 포토프린팅은 **그 세션의 열쇠**(= 공격자 열쇠)를 꺼내 제출 → 피해자 코드에 묶인 자물쇠와 **불일치**로 거부. 코드의 짝 열쇠(피해자 세션에만 있음)는 훔칠 수 없다.
- 게다가 인가 코드는 **1회용**이라, 피해자가 먼저 쓰면 이미 죽은 종잇조각.
- (같은 목적의 방어로 요청↔세션을 묶는 **`state`** 파라미터도 함께 쓴다.)

### 5. 전체 흐름 (Authorization Code + PKCE)

```
① 사용자: 포토프린팅에서 "구글 포토 연결" 클릭
② 포토프린팅: 열쇠(verifier) 생성 → 해시(challenge) 계산
   → 구글 로그인으로 리다이렉트하면서 URL에 challenge를 실어 보냄   [앞문]
③ 구글: challenge를 받아둠 (곧 발급할 인가 코드에 묶을 준비)
④ 사용자: 구글에서 로그인 + 동의 화면 [허용]                    ← 비번은 여기서만 입력
⑤ 구글 → 브라우저 → 포토프린팅: 인가 코드 전달 (토큰 아님)         [앞문]
⑥ 포토프린팅 서버 → 구글 서버: 인가 코드 + 원본 열쇠(verifier)     [뒷문]
   구글: 열쇠를 해시해 ③의 challenge와 대조 → 일치 시
   → 액세스 토큰 + 리프레시 토큰 발급                            [뒷문]
⑦ 사용자가 사진 요청 → 포토프린팅이 액세스 토큰을 들고
   구글 포토 API(Resource Server)에 요청 → 사진을 받아옴
⑧ 액세스 토큰 만료 → 리프레시 토큰으로 새 액세스 토큰 발급(재로그인 X) → ⑦ 반복
```

**진짜 토큰(⑥ 이후)은 브라우저를 한 번도 안 거친다.** 앞문엔 저가치 일회용 코드만 지나간다.

### 6. 등장인물 (4개 역할)

| 역할 | 이 시나리오에서 |
|------|----------------|
| **Resource Owner** | 사용자(나) — 데이터의 주인, 접근을 허락하는 사람 |
| **Client** | 포토프린팅 — 내 데이터에 접근하려는 제3자 앱 |
| **Authorization Server** | 구글 로그인/동의/토큰 발급 서버 — 인가 코드·토큰을 내주는 곳 |
| **Resource Server** | 구글 포토 API — 실제 데이터가 있고, 액세스 토큰을 검사해 내주는 곳 |

## 한 줄 요약

비밀번호를 직접 넘기면 (범위·취소·노출) 3대 문제가 터진다. OAuth는 비번 대신 **좁은 범위·앱별·짧은 수명의 토큰**을 내주고, 그 토큰을 **위험한 브라우저(앞문)엔 일회용 인가 코드로만, 진짜 토큰은 서버-서버(뒷문)로만** 주고받으며, 비밀키를 못 숨기는 앱은 **PKCE(세션마다 다른 열쇠/자물쇠)** 로 코드 탈취를 막는다.
