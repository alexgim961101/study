# AGENTS.md

이 저장소는 공부한 내용을 정리하는 개인 학습 저장소다. 마크다운 노트와 실행 가능한 코드 프로젝트가 함께 들어간다. 파일을 추가할 때 아래 규칙을 따른다.

## 디렉토리 구조

```
study/
├── README.md                 # 전체 목차
├── frontend/                 # 프론트엔드
├── backend/                  # 백엔드
├── infra/                    # 인프라 / DevOps
├── cs/                       # 컴퓨터 사이언스 기초
├── algorithm/                # 알고리즘 / 문제 풀이
├── languages/                # 언어 문법·특성 (java/, python/ ...)
└── projects/                 # 독립 실행 미니 프로젝트
```

최상위는 **도메인 우선**으로 나눈다. 나중에 찾을 때 제일 먼저 떠올리는 분야를 기준으로 둔다.

## 배치 규칙

### 1. 노트 = `.md` 파일 하나

- 해당 도메인 폴더 바로 아래 `.md` 파일로 둔다. 예: `backend/transaction.md`
- 한 주제의 노트가 여러 개로 늘어나면 그때 서브폴더로 승격한다. 예: `backend/spring/di.md`, `backend/spring/aop.md`
- 미리 빈 폴더를 만들지 않는다. 필요해질 때 만든다(하이브리드).

### 2. 이미지·예제 코드가 딸린 노트 = 폴더

- 이미지나 개념 설명용 짧은 예제 코드가 붙으면, 그 노트를 폴더로 만든다.
- 폴더 안에 본문을 `README.md`로 둔다. GitHub가 폴더 진입 시 자동 렌더링한다.
- 이미지는 `images/`, 예제 코드는 `code/`에 둔다.

```
backend/spring/di/
├── README.md      # 본문
├── images/
│   └── di-flow.png
└── code/          # 개념 설명용 스니펫 (독립 실행 대상 아님)
    └── ...
```

### 3. 독립 실행 프로젝트 = `projects/<이름>/`

- 자체 README·의존성·빌드 파일을 가진 완결된 프로젝트는 `projects/` 아래 둔다.
- 여러 도메인에 걸치는 풀스택 프로젝트도 여기에 둔다.
- 각 프로젝트는 자체 `README.md`로 실행 방법을 문서화한다.
- 관련 노트에서 `[chat-server](../../projects/chat-server)`처럼 링크로 연결한다.

## 네이밍

- 파일·폴더명은 영어 kebab-case를 쓴다. 예: `spring-di.md`, `url-shortener/`
- 노트 본문 내용은 한국어로 작성한다.

## languages/ 구분

- `languages/` 아래에는 그 **언어 자체의 문법·특성** 노트만 둔다. 예: `languages/java/generics.md`
- 특정 언어로 구현한 도메인 지식(예: Java의 GC를 백엔드 성능 관점에서 정리)은 해당 도메인 폴더(`backend/` 등)에 둔다. **도메인 우선.**

## 인덱스

- 루트 `README.md`에 도메인 목차를 유지한다.
- 각 도메인 폴더도 노트가 늘어나면 그 폴더의 `README.md`에 목차를 추가한다.

## 새 항목을 추가할 때 (체크리스트)

1. 어느 도메인인가? → 최상위 폴더 결정
2. 독립 실행 프로젝트인가? → `projects/<이름>/` (규칙 3)
3. 이미지·코드가 딸리는가? → 폴더로 (규칙 2), 아니면 `.md` 파일 하나 (규칙 1)
4. 영어 kebab-case로 이름 짓기
5. 한 도메인의 노트가 많아졌으면 서브폴더로 승격 검토

## 학습 방식 (개념 노트 함께 채우기)

개념 노트(`templates/concept.md` 기반)를 채울 때는 설명을 한 번에 쏟지 않고 **문답으로** 진행한다.

1. AI가 **사용자의 실제 코드에 근거한 질문**을 하나씩 던진다.
2. 사용자가 답하면 AI가 힌트·보충·교정을 한다. 틀린 답도 환영한다(이해의 실마리).
3. 개념을 작게 나눠 하나씩 쌓고, 필요하면 실습(예: 카운터)으로 직접 확인한다.
4. 개념이 충분히 쌓이면, 노트로 정리하기 전에 **AI가 사용자에게 전체 흐름을 본인 말로 설명하게 한다(teach-back).** 막히거나 어긋난 부분은 그 자리에서 보완한다.
5. teach-back으로 이해가 확인되면 AI가 **대화 내용을 노트로 최종 정리해 파일에 채운다.**
